# 基于 RK3588 AF-ALG 硬件加速的国密 SM2/SM4 软硬协同动态密钥协商与全密态传输系统

## 1. 系统架构设计

本系统采用 C/S 架构，分为客户端（Windows PC）和服务端（香橙派 5 Plus）。

### 1.1 客户端 (Windows PC)
- **角色**: 发起方，模拟终端用户。
- **核心模块**:
  - **前端 Dashboard**: 基于 Vue.js / 原生 HTML+JS，提供登录界面、状态展示、性能图表（ECharts）、密文流监控。
  - **后端服务**: 基于 FastAPI/Flask，处理前端请求，管理本地文件，发起 SM2 协商，执行 SM4 软件加密，记录性能数据。
  - **密码模块**: 纯软件实现（基于 `gmssl-python` 或 `pycryptodome`），负责 SM2 密钥对生成、协商计算、SM4 加密。
  - **通信模块**: TCP 客户端，连接香橙派。

### 1.2 服务端 (香橙派 5 Plus - RK3588)
- **角色**: 响应方，安全网关。
- **核心模块**:
  - **通信模块**: TCP 服务端，监听 PC 连接。
  - **后端服务**: 基于 Python，处理协商请求，接收密文，记录性能数据（SQLite），提供 CSV 导出。
  - **密码模块 (软硬协同)**:
    - **SM2 协商**: 软件实现（`gmssl-python`）。
    - **SM4 解密**: 硬件加速（通过 Linux AF_ALG 接口调用 RK3588 内核 Crypto 引擎），同时保留软件实现用于 Benchmark 对比。

## 2. 核心流程与协议设计

### 2.1 动态密钥协商流程 (SM2)
采用标准的 SM2 密钥交换协议，实现"一次一密"或"周期性重协商"。

1. **PC 端发起**: 生成临时密钥对 $(r_A, R_A)$，发送 $R_A$ 及自身 ID 给香橙派。
2. **香橙派响应**: 生成临时密钥对 $(r_B, R_B)$，计算共享密钥 $K$，生成校验值 $S_B$，发送 $R_B$ 和 $S_B$ 给 PC。
3. **PC 端确认**: 计算共享密钥 $K$，验证 $S_B$，生成校验值 $S_A$，发送 $S_A$ 给香橙派。
4. **香橙派确认**: 验证 $S_A$，协商完成，双方获得对称密钥 $K$。

### 2.2 全密态传输流程 (SM4)
1. PC 端选择本地文件，切片读取。
2. 使用协商的密钥 $K$ 对切片进行 SM4-CBC 加密。
3. 将密文切片通过 TCP 发送给香橙派。
4. 香橙派接收密文，调用 AF_ALG 硬件接口（或软件接口）进行 SM4-CBC 解密。
5. 解密后的明文写入本地文件。

### 2.3 通信协议帧格式 (TCP)
采用简单的 TLV (Type-Length-Value) 格式或 JSON 格式。为了方便处理二进制密文，采用自定义二进制帧头。

**帧结构**:
- `Magic` (2 bytes): 0xAA 0x55
- `Type` (1 byte): 消息类型 (0x01: 协商请求, 0x02: 协商响应, 0x03: 协商确认, 0x04: 文件传输开始, 0x05: 密文数据, 0x06: 文件传输结束)
- `Length` (4 bytes): Payload 长度 (大端序)
- `Payload` (N bytes): 消息体

## 3. 性能对比与数据记录
- **记录指标**: 传输速率 (bps)、解密延迟 (ms)、CPU 占用率 (%)。
- **对比模式**: 纯软件 SM4 解密 vs AF_ALG 硬件 SM4 解密。
- **存储**: 香橙派端使用 SQLite 记录每次传输的性能数据。
- **导出**: 提供脚本将 SQLite 数据导出为 CSV 文件。
